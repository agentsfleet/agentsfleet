# Design: Incident-recurrence fleet — the v1.0 customer wedge

Generated by office-hours on Jul 06, 2026
Status: DRAFT
Mode: Startup (pre-product)

## Problem Statement

A solo technical founder running their own production loses 1–2 hours per day to manual incident debugging: notice a symptom in the User Interface (UI), suspect "this happened before," open an ad hoc Large Language Model (LLM) session, feed it credentials from 1Password, walk three separate log surfaces, and hope the diagnosis sticks. When the same bug recurs, the entire session repeats — **nothing carries the prior resolution forward**. The core defect in the status quo is not "debugging is slow"; it is that **occurrence learning is discarded** between incidents.

agentsfleet already owns every primitive this needs — cron/webhook triggers, a per-fleet durable memory store with search (`memory.memory_entries`), an approval gate for irreversible actions, and an outbound GitHub App connector that mints short-lived tokens. The wedge is one fleet that closes the loop: **detect → match against remembered occurrences → diagnose → report to Slack → open a gated fix Pull Request (PR) → record the resolution back to memory.**

## Demand Evidence

- **Founder-lived (quantified):** ~1 incident/day; 1–2 h per LLM debugging session; the same bug later costs another 1–2 h "since there is no recollection of the bug."
- **The duct-tape twin:** Ryan Carson publicly runs this exact pipeline hand-rolled — `Production Error → Vercel Log Drain → Webhook → GitHub Actions → Amp CLI → PR + Email` — and fields "share the code" requests. Demand expressed as a workaround is the strongest pre-product signal in hand.
- **Named third-party demand (first!):** two users have asked for "injecting Slack, Notion, and agents that learn and accumulate a company brain" — demand for the *memory-accumulation* mechanism specifically. (Notion is not v1; the ask validates the architecture bet, not the v1 scope.)
- **Competitor validation:** sherlocks.ai attacks the same problem differently; adjacent public chatter (agent-driven disaster recovery playbooks, "agent auto-fixed my site during the Cloudflare outage," PostMortemGuy/BugWhisperer idea lists) confirms the space is being converged on.
- **The cautionary headline (evidence FOR the gate):** the widely-shared "AI agent deleted their production database" story is the exact failure mode the approval gate + grant-gate design prevents. Autonomy without gates is the competitor's liability and our differentiation.

## Status Quo (what the wedge replaces)

1. Symptom noticed manually (something missing in the UI, a user report).
2. Founder pattern-matches from human memory: "is this that bug from before?"
3. Manual LLM session: resolve keys from 1Password, point the model at log source #1, #2, #3.
4. 1–2 hours; resolution applied; **learning evaporates**.
5. Recurrence → repeat from step 1 at full cost.

## Target User & Narrowest Wedge

**Target user:** a solo technical founder (or first engineer at a ≤5-person team) who runs their own production, has no dedicated Site Reliability Engineer (SRE), and *already* uses an LLM ad hoc for debugging — the habit exists; the fleet replaces the manual session, it does not create a new behavior.

**Narrowest wedge (v1):** ONE fleet, ONE occurrence type — production incidents surfaced from logs.

- Trigger: cron health-check (and optionally the user's log drain POSTed to the already-shipped generic webhook trigger — zero new platform code).
- Detect: pull recent errors from the user's log sources via `http_request` + read-only `${secrets.*}` keys.
- Match: search fleet memory (`GET /v1/workspaces/{ws}/fleets/{id}/memories?query=<error signature>`) for prior occurrences — error signatures are keyword-shaped, so the shipped ILIKE search is a workable v1 matcher.
- Act: post diagnosis + prior-occurrence citation to Slack; where a fix is identified, open a PR via the GitHub connector **behind the existing approval gate**.
- Learn: record `{signature, symptom, root cause, fix PR link}` back to fleet memory so the next occurrence starts from the answer.

Explicitly NOT v1: Sentry/Notion/Zoho ingestion, cross-fleet "company brain," autonomous merge, structured platform-side matching.

## Recommended Approach

**C.** The fleet bundle ships on shipped primitives; the single platform workstream is the integration-grant read at mint + lease (M102_005 §2, carved out of the larger spec — §1 ingress, §3 Command-Line Interface (CLI), §4 docs sweep stay deferred). Rationale: an unattended, PR-opening fleet minting GitHub tokens with no grant check is the production-database-deletion headline waiting to happen; the grant table + lifecycle already exist, only the read is missing.

## Open Questions

1. Which exact log sources are the founder's "3 spots" (names + auth shape) — defines the v1 `${secrets.*}` keys and the SKILL.md walk order.
2. Memory key convention for occurrence records (e.g. `incident:<signature-hash>`) — needs to be stable enough for ILIKE recall.
3. Match-confidence threshold: when does the fleet open a PR vs. report-only? (v1 default: report always; PR only when the SKILL cites a prior occurrence with a fix link.)
4. M102_005 carve: does §2 become its own workstream (M102_006?) or does M102_005 slim down in place? Also fold in the two missing prerequisites found this session (App webhook secret field, `installation_id → workspace_id` table) when §1 is eventually picked up.
5. Zoho Recruit vs Zoho Desk OAuth scope compatibility — deferred with the digest fleet.

## Success Criteria

- Seeded-recurrence drill: fleet detects a re-injected known error and its Slack report cites the prior occurrence + fix PR.
- A real second-occurrence bug goes symptom → gated PR in < 15 minutes of wall clock vs the 1–2 h manual session.
- Negative test: a fleet without an approved `github` grant gets `grant_required` at mint — no token, no upstream call.
- One external design partner (not the founder) connects their log source and runs the loop on their stack.

## Dependencies

- Grant-gate workstream (M102_005 §2 carve-out) — the only platform blocker.
- Founder's log sources reachable via read-only HTTP APIs (vault-stored keys).
- GitHub App connected + grant approved for the fleet's workspace; Slack connector for report delivery.

## Follow-on fleets (sequenced, not v1)

1. **Slack outage-watcher** — shares this fleet's DNA; Slack inbound ingress already shipped.
2. **PR-review fleet** — blocked on the GitHub App-webhook ingress decision (M102_005 §1 + repo-routing question).
3. **PR-fix-and-ship loop** — the wedge fleet's PR-opening arm, generalized.
4. **Zoho Recruit end-of-day digest** — pending the Recruit-vs-Desk OAuth check.

## What I noticed

The founder said "I find myself clueless — I go to 3 spots and tell the LLM to find the cause, and it still may linger" — lived pain, precisely located. The scope grew twice mid-session (logs → Sentry → Slack → Zoho; "tell me" → "solve it by a PR") and both times the founder accepted the narrowing without losing the vision — the generalization is parked as architecture, not deleted. On "assist first vs act," the founder overrode the safe default with an unambiguous "I WANT TO ACT ON IT" — conviction, recorded, and the design honors it with the gate as the counterweight.
